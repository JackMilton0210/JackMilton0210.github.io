Beat 'Em Mop

January - April 2024

itch.io

Beat Em Mop is a 2D arcade style fighting game where you play as the janitor who has had enough of his job and decides to take out his rage on the co-workers around him all while keeping the place spotless.

Beat 'Em Mop was developed as part of a 7-person group project over the course of 4 months. I was the game's programmer and was responsible for coding, implementing assets and managing source control.

The biggest challenge through the project was the combat system.

The earliest implementation saw the player simply damage enemies if they were standing in front of him. This was sufficient for the beginning stages of development.

Once the first pass animations were implemented, a key frame system was developed such that the attack hitbox could be repositioned and resized for each frame. The hitboxes were then mapped to match the weapon perfectly.

However after the final versions of the animations were developed, it became apperent that this could be improved. The keyframes were extended farther to provide a more fun and fluid feeling.

Spaceship Shooter

April 2022

SpaceshipShooter.zip
Combat System

Each ship was going to be given a precise collision box made up of multiple rectangles. Once the movement and combat systems had been designed this became an unrealistic goal to complete within the given timeframe. Each rectangle would need to have its position and rotation recalculated each frame and the number of collision checks to be performed each frame would be unmanageable. Instead each ship was a given a bounding circle. The position was static and the rotation was (obviously) irrelevant. If I were to make something similar now, I would implement a Bounding Volume Hierarchy which would keep the number of collision checks low while allowing for greater accuracy.

Movement System

The first iteration had the player control the angle of the ship directly, turning instantly on key press. This movement was passable but felt too snappy and did not make the ship appear as though it were in a vaccuum

In an attempt to solve this the second iteration saw the player control the throttle and angle of the engine instead. The game would then calculate the new velocity of the craft according to the effect the engine should have. This involved a large redesign of some core components of the game as well as modifying art assets so that the ship and engine were seperate sprites but effectively made the ship behave more realisticly. This new system was hard to understand as a player. Controlling the ships speed through acceleration was close to impossible and the ship would be unable to turn at low speeds due to having less angular momentum which was completely unintuitive.

The final version of movement involved a blend of the two sytems where the player would control the engine directly but the movement of the ship was not wholey based on physics. If the engine is full throttle, facing right then the ship turns quickly to the right.

Wemos - Simon Says

December 2021

This was my introduction to working with hardware, something I would like to explore in the future. Here I created a version of the classic "Simon Says" game where a sequence of lights will flash and the user must repeat the same pattern back.

The rotary encoder was used to change the difficulty of the game, making the duration that the lights flash shorter at higher levels. Each light was also given a note to play when flashing to help distinguish between them.

This was one of my first projects using C++ and it shows. Many code blocks are inefficient and poorly structured. The entire thing is also written in a single text file making it very difficult to read.
